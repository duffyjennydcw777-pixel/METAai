# 🔮 AI Prompt Library — 25 промптов для разработки

> Копируй и вставляй. Каждый промпт проверен на реальных проектах.

---

## 📐 Архитектура

### ARCH-01: Спроектируй архитектуру
```
Спроектируй архитектуру для [описание проекта].
Требования: [список требований].
Стек: [технологии].

Формат ответа:
1. Компоненты и их роли (таблица)
2. Диаграмма взаимодействия (mermaid)
3. Структура файлов
4. Ключевые ADR (минимум 3 решения с обоснованием)
5. Риски и как их митигировать
```

### ARCH-02: Оцени решение
```
Оцени это архитектурное решение:
[описание решения]

Контекст: [что строим, какой масштаб]

Формат ответа:
- Плюсы (конкретные, не общие)
- Минусы (конкретные, не общие)
- Альтернативы с обоснованием
- Рекомендация: использовать / не использовать / модифицировать
```

### ARCH-03: Декомпозиция задачи
```
Декомпозируй эту задачу на подзадачи:
[описание задачи]

Для каждой подзадачи укажи:
1. Что делать (конкретно)
2. Какие файлы менять
3. Code Complexity Level (1/2/3)
4. Зависимости от других подзадач
5. Оценка времени
```

## 🔍 Code Review

### REV-01: Review diff
```
Сделай code review этого diff:
[diff]

Проверь:
1. Логические ошибки
2. Security (SQL injection, XSS, secrets в коде)
3. Performance (N+1, лишние запросы, утечки памяти)
4. Стиль и конвенции
5. Edge cases

Формат: 🔴 Critical / 🟡 Warning / 🟢 Good для каждого пункта.
```

### REV-02: Найди баг
```
В этом коде есть баг. Симптом: [описание].

Код:
[код]

1. Найди root cause
2. Объясни почему это происходит
3. Дай исправленный код (не объяснение, а КОД)
4. Предложи тест, который бы поймал этот баг
```

### REV-03: Оптимизация
```
Оптимизируй этот код:
[код]

Контекст: [нагрузка, данные, SLA]

1. Что сейчас плохо (с замерами/оценками)
2. Оптимизированная версия (полный код)
3. Что стало лучше (конкретные цифры)
4. Trade-offs (что потеряли)
```

## 🧪 Тестирование

### TEST-01: Генерация тестов
```
Напиши тесты для этого кода:
[код]

Code Complexity Level: [2/3]
Фреймворк: [pytest / jest / go test]

Обязательно:
- Happy path
- Error cases (невалидный ввод, null, пустые данные)
- Edge cases (граничные значения, concurrent access)
- Mock внешних зависимостей
```

### TEST-02: Тест-план Level 3
```
Составь тест-план для Level 3 (Complex) задачи:
[описание задачи]

Включи:
1. Unit tests (список)
2. Integration tests (с реальными зависимостями)
3. Regression tests (что НЕ должно сломаться)
4. Stress tests (если applicable)
5. Dry-run план
6. Post-deploy мониторинг
7. Rollback план

Для каждого теста: что проверяем, expected result, priority.
```

## 🚀 Деплой

### DEP-01: Pre-deploy checklist
```
Проверь готовность к деплою:

Проект: [название]
Изменения: [описание]

Checklist:
1. [ ] Все тесты проходят
2. [ ] .env синхронизирован (local vs production)
3. [ ] Нет hardcoded secrets
4. [ ] CHANGELOG обновлён
5. [ ] Нет console.log / print debugging
6. [ ] Миграции БД готовы (если есть)
7. [ ] Rollback план есть
8. [ ] Мониторинг настроен

Покажи конкретные проблемы, если есть.
```

### DEP-02: Deploy script
```
Создай deploy скрипт для:
Проект: [название]
Сервер: [IP:PORT]
Стек: [технологии]
Сервис: [systemd / docker / pm2]

Скрипт должен:
1. Залить файлы (scp/rsync)
2. Установить зависимости
3. Запустить миграции (если есть)
4. Перезапустить сервис
5. Проверить health
6. Откатить при ошибке
```

## 💡 Рефакторинг

### REF-01: DRY/SRP рефакторинг
```
Проведи рефакторинг этого кода:
[код]

Фокус: [DRY / SRP / Naming / Simplification]

1. Что не так (конкретные code smells)
2. Рефакторенный код (полностью, не фрагменты)
3. Что улучшилось (метрики: строки, сложность, coupling)
```

### REF-02: Миграция API
```
Помоги мигрировать с [старого API/библиотеки] на [новый].

Текущий код:
[код]

1. Маппинг: старый метод → новый метод
2. Breaking changes
3. Мигрированный код
4. Тесты для проверки миграции
```

## 📝 Документация

### DOC-01: README генерация
```
Сгенерируй README.md для проекта:
Название: [название]
Описание: [описание]
Стек: [технологии]

Включи:
- Badge'ы (build status, version)
- Описание (2-3 предложения)
- Quick Start (5 минут до рабочего проекта)
- Архитектура (mermaid диаграмма)
- API Reference (если есть)
- Contributing guide
```

### DOC-02: API документация
```
Задокументируй эти endpoints:
[код endpoints]

Формат: OpenAPI-style markdown
Для каждого endpoint:
- Method + Path
- Описание
- Parameters (query, body, path)
- Response (200, 400, 401, 500)
- Пример запроса (curl)
- Пример ответа (JSON)
```

## 🛡️ Security

### SEC-01: Security audit
```
Проведи security audit этого кода:
[код]

Проверь:
1. SQL injection
2. XSS
3. CSRF
4. Secrets в коде
5. Auth bypass
6. Rate limiting
7. Input validation
8. Error messages (не раскрывают внутренности?)

Для каждой найденной проблемы:
- Severity (Critical/High/Medium/Low)
- Где именно
- Как эксплуатировать
- Как исправить (код)
```

---

> 💡 **Совет:** комбинируй промпты. Например, ARCH-03 (декомпозиция) → TEST-01 (тесты для каждой подзадачи) → DEP-01 (pre-deploy).
