# 🎯 CODE COMPLEXITY LEVELS — Стратегия тестирования AI

> AI ОБЯЗАН классифицировать КАЖДОЕ изменение кода перед деплоем.
> Уровень определяет объём тестирования. Нарушение = ПРОВАЛ.

---

## 🟢 Level 1 — Trivial (Деплой без тестов)

**Критерии**: изменение НЕ может сломать логику приложения.

| Тип изменения | Примеры |
|---------------|---------|
| Конфиги (визуальные) | `.gitignore`, комментарии в `.yml`. **⚠️ Конфиги, меняющие поведение (порты, URL, feature flags) = Level 2** |
| Документация | README, CHANGELOG, `docs/`, комментарии в коде |
| Статический текст | Тексты кнопок, сообщений бота, UI-лейблы |
| Стили | CSS, цвета, шрифты, отступы |
| Логирование | Добавление `logger.info()`, `print()` для отладки |
| Простые переименования | Переменные, файлы (без изменения логики) |

**Действие**: Деплой сразу. Тесты не нужны.

---

## 🟡 Level 2 — Standard (3 целевых теста)

**Критерии**: изменение затрагивает логику, но в ограниченном scope.

| Тип изменения | Примеры |
|---------------|---------|
| Новые функции/эндпоинты | Простой CRUD, новая команда бота |
| Баг-фиксы | Чёткий scope, понятная причина |
| Новые поля моделей | Добавление колонки без миграции данных |
| Простые рефакторы | Извлечение функции, перенос модуля |
| Форматирование данных | Парсинг, сериализация, шаблоны |
| Фильтры и валидации | Новые проверки на входные данные |

**Действие**: 3 теста минимум:
1. ✅ **Happy path** — основной сценарий работает
2. ❌ **Error case** — обработка ошибок/невалидных данных
3. 🔄 **Edge case** — граничные условия (пустые данные, null, overflow)

> **Когда 3 теста НЕ достаточно** (обосновать в ответе):
> - Функция имеет >3 ветвлений (if/elif/else) → +1 тест на ветку
> - Функция работает с несколькими типами данных → +1 тест на тип
> - Функция вызывает внешний сервис → +1 тест на mock

---

## 🔴 Level 3 — Complex (Тестить до 85% уверенности)

**Критерии**: изменение может привести к потере данных, денег или пользователей.

| Тип изменения | Примеры |
|---------------|---------|
| Финансовая логика | Платежи, подписки, биллинг, расчёт цен |
| Аутентификация | Токены, сессии, права доступа, RBAC |
| Миграции БД | ALTER TABLE на production данных |
| Асинхронный код | asyncio, параллельные задачи, race conditions |
| Мульти-сервисные цепочки | API → DB → Telegram → X-UI |
| Безопасность | Шифрование, VLESS-ключи, OAuth, JWT |
| Core-алгоритмы | Скоринг, матчинг, маршрутизация, дедупликация |
| Инфраструктура | CI/CD, Nginx, Docker, systemd, cron |
| Работа с user data | Массовые обновления, удаления, миграции пользователей |

**Действие**: Тестировать до достижения **85% уверенности**:
1. Все тесты Level 2 (happy + error + edge)
2. **Integration test** — проверка взаимодействия компонентов
3. **Regression test** — старый функционал не сломан
4. **Stress/boundary test** — большие объёмы, таймауты, concurrent access
5. **Dry-run на staging** — перед production (если применимо)
6. **Ручная верификация** — просмотр логов после деплоя

> **Как измерить 85%**: AI должен озвучить оставшиеся риски (15%).
> Формат: "Уверенность 85%. Риски: [список]. Рекомендация: [деплоить / ждать]"

---

## 🏷️ Как классифицировать

При КАЖДОМ изменении кода AI указывает в ответе:

```
📊 Complexity: Level [1/2/3] — [Trivial/Standard/Complex]
🧪 Testing: [описание плана тестирования]
```

**Правило повышения**: Если есть сомнения между двумя уровнями → выбрать ВЫСШИЙ.

---

## 📋 Примеры классификации

| Задача | Level | Обоснование |
|--------|-------|-------------|
| Обновить текст приветствия бота | 🟢 1 | Статический текст, логика не затронута |
| Добавить endpoint `/api/users` | 🟡 2 | Новая функция, ограниченный scope |
| Изменить логику скоринга заказов | 🔴 3 | Core-алгоритм, влияет на все уведомления |
| Добавить поле `phone` в модель User | 🟡 2 | Новое поле, без миграции данных |
| Миграция пользователей на новую ноду VPN | 🔴 3 | User data + внешний API + production |
| Исправить опечатку в CHANGELOG | 🟢 1 | Документация |
| Рефакторинг subscription gating | 🔴 3 | Биллинг + доступ пользователей |

---

## 🔄 Обновление

- При обнаружении нового паттерна → добавить в соответствующий уровень
- При false-negative (баг проскочил через Level 1/2) → повысить уровень для этого типа
- При false-positive (избыточное тестирование) → обосновать понижение

---

## 🚨 Post-Mortem Protocol

Если баг проскочил мимо классификации:

1. **Немедленно**: записать в `anti-patterns.md` (код CCL-XXX)
2. **Повысить уровень**: тип изменения, вызвавший баг, повышается на 1 уровень навсегда
3. **Обновить таблицу примеров**: добавить реальный кейс с пояснением "почему недооценили"
4. **Формат записи**:
   ```
   CCL-001: [Тип изменения] был классифицирован как Level [N], вызвал [описание бага].
   Новый уровень: Level [N+1]. Причина недооценки: [причина].
   ```
