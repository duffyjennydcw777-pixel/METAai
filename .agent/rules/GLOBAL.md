# 🌍 GLOBAL RULES — Обязательны для ВСЕХ проектов

> Эти правила загружаются AI-агентом ПЕРВЫМИ при каждом ответе.
> Нарушение = ПРОВАЛ.
> Связанные файлы: `MODES.md`, `CODE_COMPLEXITY.md`, `CONVENTIONS.md`, `MAKER_PROFILE.md`, `PROJECT.md`.

---

## 🧠 Контекст и память

1. **Knowledge Items First**: В начале КАЖДОЙ беседы проверить KI. Особенно: `ai-anti-patterns`, `maker-profile-global`, `project-management-patterns`, `deployment-patterns`.
2. **session_state.md**: Каждые 15-20 сообщений обновлять в scratch-директории. Фиксировать: задачу, решения, данные, TODO. При потере контекста — перечитать.
3. **Anti-Pattern Registry**: ПЕРЕД деплоем, SSH, Python/datetime кодом — сверяться с `knowledge/ai-anti-patterns/artifacts/anti-patterns.md`.
4. **MODES.md**: AI ОБЯЗАН загружать режимы мышления из `.agent/rules/MODES.md`. Указывать текущий режим в ответе: `🎭 Режим: [название]`.
5. **REFLEXION_LOG.md**: При стратегических решениях (Шахматы/Штаб/Торговец) — сверяться с `REFLEXION_LOG.md`. Не повторять стратегические ошибки.

## 📋 Документация

4. **CHANGELOG.md**: Проверять наличие в каждом проекте. Задача НЕ завершена пока CHANGELOG не обновлён.
5. **docs/ (Documentation 2.0)**: Если проект серьёзный — должна быть папка `docs/` с 4 столпами:
   - `1_PRODUCT.md` — что, для кого, фичи
   - `2_ARCHITECTURE.md` — стек, структура, ADR
   - `3_INFRASTRUCTURE.md` — серверы, порты, домены (БЕЗ паролей!)
   - `4_DEPLOY_RUNBOOK.md` — пошаговый деплой
6. **DECISIONS.md**: ПЕРЕД предложением архитектурного решения — проверить Decision Log. Не предлагать то, что уже отвергнуто.
7. **SOLUTION_PATTERNS.md**: ПЕРЕД написанием кода — проверить библиотеку паттернов. Если есть PAT-XXX — использовать.

## 🔒 Безопасность

6. **Секреты**: НИКОГДА не оставлять API-ключи, пароли, токены в коде или Markdown. ТОЛЬКО `.env`.
7. **chmod/chown**: После КАЖДОГО деплоя файлов на Linux — напоминать о правах.
8. **SSH**: ВСЕГДА `-p 2222`. ТОЛЬКО по IP, не по домену. Сверяться с таблицей серверов (SRV-001).
9. **sed через SSH из PowerShell**: ЗАПРЕЩЕНО. Создать Python-скрипт.

---

## 🏗️ Качество

10. **Zero Copy-Paste**: Если AI может создать файл, прописать конфиг, сгенерировать команду — делать самому. Не заставлять человека копировать.
11. **НЕ ЛОМАТЬ**: При обновлении кодовой базы — 100% уверенность, что ничего не сломано.
12. **Git-дисциплина**: После 5+ изменённых файлов — напоминать о коммите. Не уходить из сессии с незакоммиченными изменениями.
13. **Премиум, не MVP**: Создатель требует премиум-качество. Никаких минимальных поделок.
14. **Code Complexity Levels**: ПЕРЕД деплоем классифицировать изменение (Level 1/2/3). См. `CODE_COMPLEXITY.md`. Level 1 = без тестов, Level 2 = 3 теста, Level 3 = тестить до 85% уверенности.
15. **Multi-Agent Pipeline**: Для Level 2+ — запустить `python review.py review --level N`. Агенты проверят код автоматически.

---

## 🕵️ Протокол расследования (Detective-First)

16. **При баге — СНАЧАЛА режим Детектив.** НЕ переписывать код до проверки гипотез. Порядок:
    ```
    Факты → Гипотезы (ранжированные) → Проверка главной гипотезы → Минимальный фикс → Регрессия
    ```
17. **Запрещено при дебаге:** переписывать файлы без гипотезы, менять несколько вещей одновременно, игнорировать логи.
18. **Клиника vs. Детектив:** Детектив = конкретный баг. Клиника = системная нестабильность. Разные протоколы.

## 🔀 Model Routing (Token Efficiency)

19. **Авто-рекомендация модели**: В начале КАЖДОГО ответа AI оценивает сложность запроса и текущую модель. Если модель избыточна или недостаточна — рекомендовать переключение:
    ```
    🔀 Model: ✅ Opus (оптимально для этой задачи)
    🔀 Model: ⚡ Переключи на Flash — это Level 1 задача
    🔀 Model: ⬆️ Переключи на Opus — это Level 3, нужна тяжёлая артиллерия
    ```
20. **Маршрутизация по CCL**:
    - **Level 1** (доки, CSS, конфиги, "найди файл") → Gemini Flash
    - **Level 2** (новые фичи, баг-фиксы, рефакторы, скрипты) → Gemini Pro / Claude Sonnet
    - **Level 3** (архитектура, мета-инженерия, мульти-файловые изменения, security, стратегия) → Claude Opus
21. **Правило экономии**: Opus НЕ должен использоваться для: обновления CHANGELOG, правок README, простых команд git, однострочных фиксов. Это Flash/Pro задачи.
22. **Правило эскалации**: Если задача оказалась сложнее, чем ожидалось — рекомендовать повысить модель, а не мучиться на слабой.
